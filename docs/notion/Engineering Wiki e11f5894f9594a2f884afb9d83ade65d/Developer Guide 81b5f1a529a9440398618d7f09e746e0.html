Developer Guide

This documentation will include Cloud9 developer guide when contributing to the repository. One must read the entire documentation to make sure that he or she understands what are the expected behaviors we are expecting when developers contribute to the code base.

GitFlow

Our development will be based off of branching model for GitFlow, created by Vincent Driessen, and adapted to our own development process.

Key Benefits of GitFlow

Parallel Development

One of the great things about GitFlow is that it makes parallel development very easy, by isolating new development from finished work. New development (such as features and non-emergency bug fixes) is done in feature branches, and is only merged back into main body of code when the developer(s) is happy that the code is ready for release.

Although interruptions are a BadThing(tm), if you are asked to switch from one task to another, all you need to do is commit your changes and then create a new feature branch for your new task. When that task is done, just checkout your original feature branch and you can continue where you left off.

Collaboration

Feature branches also make it easier for two or more developers to collaborate on the same feature, because each feature branch is a sandbox where the only changes are the changes necessary to get the new feature working. That makes it very easy to see and follow what each collaborator is doing.

Developer Guide

Things to remember

  • Feature branches are created from dev branch :)
  • Create small and specific branches.
  • Constantly pull dev to stay up to date with other people's changes.
    • You can do this by:
      1. Checking out dev:
        • git checkout dev
      1. Pulling any new changes from the remote:
        • git pull <remote>
      1. Returning to your branch (checkout your branch again):
        • git checkout [your branch]
      1. Merging dev into your branch:
        • git fetch
        • git pull origin dev

  • The dev branch will constantly be pulling from master, so testing your code on dev should prevent bugs from getting on master.
    • Once you have tested your code on the dev branch, create a pull request (PR) to merge your branch into dev.
      • Who and how many people you send it to should depend on the significance of your code.
      • Write a short (1-4 sentences) description of your PR.
      • You can send your PRs to other people through Slack.

    • Create a branch for any change that you want to make.

Naming Branch

  • Feature branch: feature-*
    • Ex: feature-migrate-firebase
  • Fixes branch: bugfix-*
    • Ex: bugfix-package-dependencies
  • Testing branch: testing-*
    • Ex: testing-firebase-authentication
  • Build process branch: build-*
    • Ex:build-add-testing-workflows
  • Documentation branch: document-*
    • Ex: document-add-readme
  • Clean up / Refactor branch: cleanup-*
    • Ex: cleanup-remove-unused-function

Development workflow and branch lifecycle:

  1. You want to make a change to the codebase.
  1. Decide which features you want to contribute to and name it with a descriptive branch name.
  1. Create a new branch off of dev.
  1. Make changes to your branch.
  1. Finish making changes to your branch.
  1. Merge your branch into dev and ensure that everything works. If you run into merge conflicts, carefully resolve them. If you are unsure how to resolve them, ask someone else for help. Do not resolve merge conflicts if you do not understand how to resolve merge conflicts or if you do not understand the code that you are deleting.
  1. Your branch has served its purpose and it has been merged into dev. Your branch should not be used anymore. To enforce this, after branches are merged, they are deleted from the remote repository.

Code Review

Each sub-team should have at least one member from it to review pull request.

We required at least 3 members to approve the pull request so please try to do the code review as often as you can.

  • Questions to think about when doing code reviews
    • Does this function does what I expect it to do?
    • Do I understand the implementation?
    • Does this code achieve the requirements?
    • Is the coding style consistent with the requirements?
  • Pull and Test it out, see if everything works as normal
  • Don't spend too much time going through everything single line
  • Give helpful feedback for improvements or what was done well
  • Show respect when giving feedback
